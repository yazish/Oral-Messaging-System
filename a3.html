Peer to Peer - Distributed Consensus System

In this assignment, you will create a node for a peer-to-peer consensus system implementing a distributed database of words, using the Oral Message solution to consensus. For this assignment, we will use the push model described (and acted out) in class.

In order to initially connect to your peers, your not will present itself using gossiping: sending a GOSSIP message to two of a set of well known peers. This will get propagated across the network, with new peers reaching out to your node to establish a mesh network. After joining the network, your node will continue use gossiping as a heartbeating mechanism to notify the other peers that it is still active.

The OM Peer-to-Peer Protocol

The protocol has three main components, the message format and transport protocol, the gossiping method, and the consensus algorithm.

Messages and Transport

Peers will communicate with each other via UDP messages. Message will be JSON formatted and text encoded. Unlike TCP messages, UDP message cannot be partially read from the socket by a call to read(), they will always either be there in full, or not. To simplify things a bit, you can assume that all peer messages will be less that 4096 bytes in length.

Gossiping

Your node will announce a GOSSIP message to a peer in the mesh network to enter, and will continue to do so to stay active in the network. Upon receiving a GOSSIP message, your node must forward it (unmodified) to up to 5 randomly chosen known peers to continue along the mesh. In order to ensure that the same messages aren't forwarded around the peer group forever, each GOSSIP message has an id field containing a unique string. Keep track of recent gossips you've received, and ignore them if you receive them again!

Other Notes

  • If you receive a GOSSIP from a peer previously unknown to you, reply to them with a GOSSIP_REPLY.
  • Anytime you receive either a GOSSIP or a GOSSIP_REPLY from an unknown peer, make sure to add them to a list of known peers!
  • GOSSIP to up to 5 random known peers every minute as a heartbeat. *** Drop any nodes** from your list of peers that have not sent a GOSSIP or GOSSIP_REPLY for 2 minutes.

Consensus

Once your node has gossiped and joined the peer network, you are able to begin taking part in consensuses! The goal of this is to simulate a distributed database. In the real world this could be any sort of data that a system needs to keep consensus over. (Key-value pairs anyone?) In our case, each peer will keep track of a simple list/array of 5 words as their database. Consensus will be used to propose and agree on new values for each of the 5 indexes in this database.

As mentioned previously, you'll be implementing the Oral Message protocol. To keep it straightforward, we're going to use the same "push" model that we demonstrated in class. In the original paper, the recursive algorithm is described from the base case up. While that's great for proving correctness, it's not how you'll actually need to implement it, so here's a top down overview of how the algorithm works, again, just like we did in class:

Initiating a Consensus

When a consensus is first started, the root of a consensus tree, the initiator (commander/general) should calculate the maximum level of fault-tolerance mmm that can be achieved given the peers they are aware of. An OM(m)OM(m)OM(m) consensus message is created with the following:

  • The fault-tolerance level mmm
  • A proposed database index and value, a
  • A list of all peers that should take part in the consensus
  • A unique ID for the consensus round
  • An empty/null "parent" field to indicate that this is root consensus

This message should be sent to all the peers in the list.

Note: At this point, the initiator can simply commit the new value to their database! Remember that OM(m)OM(m)OM(m) assumes that there are mmm faults/liars at most. If there are more faults, we wouldn't achieve consensus anyways, so the proposer can choose to believe whatever they like!

Receiving a Consensus Message

Receiving a consensus message means that someone has initiated a new consensus with the peer. If the consensus has a parent, it is a sub-consensus. Otherwise, it is the root consensus.

If m>0m > 0m>0...

...then this is a message the peer can't totally trust and needs to validate! In order to do so, the peer need to run a sub-consensus with the other peers to make sure that the initiator isn't lying to them all. This looks similar to initiating a root consensus, but with the following changes:

  • The new fault tolerance level is m−1m-1m−1
  • The list of peers is the same as the parent with the new initiator removes
  • The "parent" field should be set to the ID of the consensus that triggered the sub-consensus

This new sub-consensus message should then be sent to all the relevant peers to kick it off.

If m=0m = 0m=0...

...then this is is a "leaf" consensus and doesn't justify further validation. (The consensus can't handle more faults, so no point in sending more messages.) In this case, the recipient can assume that this is the honest value that the sender received from the initiator of the parent consensus, and can record it as such. At this point, the recipient heck to see if the parent consensus is "complete", and if so determine what the majority of the peers heard from the initiator and determine what to do with that value:

  • If the parent consensus has a parent of its own, then recursively record the result of the newly completed sub-consensus up the tree
  • If the parent consensus is the root (ie. has no parent of its own), then the peers have all achieved full consensus!

Monitoring/Controlling Your Node

Your node should be able to be interacted with in one of two ways in order to get information about its state and to modify its behaviour: a TCP-oriented CLI, or a webserver interface.

TCP CLI

For this option, you can write a small Python program or use telnet as the CLI for interaction.

Your CLI must accept the following commands:

  • peers - list all known peers, with last words received from each.
  • current - the current word list.
  • consensus # <text> - where # is a numeric index. Run a consensus to set the database entry at the given index to the given word.
  • lie - begin lying. Lie either all the time, or a set percentage of the time.
    • You may want to provide the word, choose a random word, or always choose the same incorrect word to lie with.
  • truth - stop lying. Always return 'honest' answers.
  • exit - close the command-line interface.

Webapp Interface

It must support displaying the same data and doing the same tasks as described for the CLI above (lie/set/do consensus).
